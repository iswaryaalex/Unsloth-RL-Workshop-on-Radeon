Why Local AI

What would you finetune
if your data never left your machine?

Most powerful AI models live in the cloud — which means your data has to leave your hands. Local finetuning removes that constraint entirely. The real unlock isn't just privacy — it's what becomes possible when the barrier disappears.

🏥

Healthcare

Patient records, clinical notes, diagnostic history. A model finetuned on your hospital's data could assist documentation, triage, or research — without a single record leaving your network.

🏦

Finance

Transaction history, risk profiles, proprietary strategies. Compliance regulations make cloud AI a non-starter for most financial data — local models change that equation entirely.

🧬

Personal Wellbeing

Sleep, mood, nutrition, fitness logs. An AI trained on your actual patterns — not population averages — gives genuinely personalized health guidance, not generic advice.

🎯

Personalization

Your communication style, preferences, context, history. A model trained on your own data is fundamentally different from a general assistant — it's genuinely, deeply yours.

🔁

Continuous Learning

Cloud models are static snapshots. With local finetuning, your model can keep learning — from new data, new feedback, new domains — without waiting for a vendor's next release cycle.

🏛️

Institutional Knowledge

Every organization has tacit knowledge that lives in documents, emails, and conversations. Local finetuning lets you encode that knowledge into a model that understands your specific context.

The common thread: trust. Every one of these use cases shares the same blocker — you can't send the data to a third party. Healthcare can't send patient records to OpenAI. Finance can't send trading strategies to Anthropic. Local finetuning isn't just a technical choice — it's what makes AI deployable in the real world, in regulated industries, in sensitive contexts, at the institutional level. Unsloth × AMD

Finetuning techniques —
2× faster, 70% less VRAM

Unsloth uses custom Triton kernels and mathematical optimizations to make finetuning dramatically more efficient — with no accuracy loss. On AMD hardware, the same gains apply.

Core technique

LoRA / QLoRA

Instead of updating all weights, LoRA adds small trainable adapter matrices to key layers. 60–80% VRAM reduction vs full finetuning. Unsloth further optimizes LoRA backward passes with custom kernels.

Memory

Gradient Checkpointing

Rather than storing all intermediate activations, Unsloth recomputes them during the backward pass. Trades compute for memory — essential for long-context models on limited VRAM.

Precision

16-bit Training

On AMD hardware, Unsloth defaults to BF16 / FP16 for stable mixed-precision training. Near full-precision quality at half the memory footprint — the standard for modern LLM finetuning.

Reinforcement learning

GRPO

Group Relative Policy Optimization. More memory-efficient than PPO, no separate critic model needed. Scores groups of completions and updates the policy toward higher-reward outputs.

Reinforcement Learning

Teaching a model to reason,
not just memorize

Supervised finetuning teaches a model to imitate — show it examples, it learns to copy. That works for style and format, but the model memorizes patterns rather than developing strategy.

Reinforcement Learning flips the paradigm. Instead of showing the model the answer, you give it a reward signal for producing good outputs. It explores, makes mistakes, and gradually learns to maximize reward.

We use Sudoku as the RL environment because it has a fully verifiable, objective solution. No human ratings, no fuzzy heuristics — we check the constraints programmatically and compute a precise reward score.

Gemma 4 E2B starts with zero Sudoku ability. After GRPO training it develops genuine solving strategies — not by memorizing puzzles, but by learning what makes a valid grid.

The GRPO training loop

1
Environment sends a puzzle

Model receives an unsolved 9×9 grid as the input prompt.

2
Model generates completions

Gemma 4 E2B produces a filled grid inside <answer> tags. Multiple completions sampled per prompt.

3
Reward functions score each one

Format, digit validity, row, column, and box uniqueness — combined into a scalar reward.

4
GRPO updates the weights

High-reward completions are reinforced relative to the group average. Poor strategies are discouraged.

The Sudoku environment

A Sudoku puzzle is the perfect RL toy problem. Rules are unambiguous — each row, column, and 3×3 box must contain digits 1–9 exactly once.

The reward signal is fully verifiable. We check every constraint programmatically. This makes it ideal for GRPO where clean reward signals are essential.

Watching the reward curve climb over training steps is a direct window into the model learning to reason — not memorize.

Reward Engineering

The reward function is
the real design problem

In RL finetuning, you define what counts as a good solution — the model figures out how to get there. Getting the reward function right is the most creative part of the process.

📐

Format Reward

Did the model output the answer in the correct structure? Before checking correctness, output must be parseable. Trains the model to follow instructions reliably.

def format_reward(completion): if "<answer>" in completion \ and "</answer>" in completion: return 1.0 return 0.0
🔢

Digit Validity

Every cell must contain exactly one digit 1–9. A basic sanity check that the model understood the domain constraints of the task before we check anything else.

def digit_reward(grid): for row in grid: for cell in row: if cell not in "123456789": return 0.0 return 1.0
↔️

Row / Col / Box

Partial credit per valid row, column, and 3×3 box. Dense rewards give the model a gradient signal even when only some constraints are satisfied — critical for learning.

def row_reward(grid): score = 0 for row in grid: if len(set(row)) == 9: score += 1 return score / 9.0
Dense vs sparse rewards. If you only reward the model when the full Sudoku is correct, it gets zero signal early in training — nothing to learn from. Dense rewards give a gradient at every step: even 3 valid rows scores 0.33, and the model knows it's heading in the right direction. This is the key insight that makes RL finetuning practical on real tasks.
